# 2026-05-28 V1 组会同步

## 会议目标

对齐 Greyfield Next V1 当前产品进度、剩余缺口，以及新增 2 位同事后可以拆分的独立模块。

## 当前结论

V1 的桌宠底座已经成立：透明桌面宠物窗口、真实 Live2D 加载、模型命中、拖动、缩放、穿透、文字输入输出、Stop、最近上下文持久化、provider 错误恢复和 CI 保护都已经有测试或 harness 证明。

现在还不能宣布 V1 完成。剩余主要不是“能不能跑”，而是把用户直接看到的聊天、设置、气泡边缘场景和后续语音能力打磨到可交付状态。

## 还差什么

### P1：用户直接看得到的 polish

1. 气泡 QA 收尾：
   - 右侧屏幕边缘翻转和 viewport clamp；
   - 气泡开关后输入 shape add/remove；
   - 开关后透明区域命中仍然透明。
   - 当前状态：`E:\greyfield-next-bubble-qa` / `codex/v1-bubble-toggle-qa` 已开始实现，但尚未完成验证和 PR。

2. Chat 窗口 polish：
   - Stop 在 streaming 时始终明显；
   - thinking / speaking / error 状态清晰；
   - 错误后用户能自然看到“可以修改并重试”；
   - 长回复完整留在 Chat，宠物气泡只做短提示。

3. Settings polish：
   - provider 配置状态更像产品，而不是表单调试页；
   - Test LLM 的进行中、成功、失败、被拒绝状态更清楚；
   - 模型路径、角色文件、API key 保存状态不让普通用户困惑。

### P2：真实文字聊天可交付

1. OpenAI-compatible 文本链路已经可演示，但还需要把真实 provider 配置、错误恢复、Test LLM 引导打磨成普通用户能理解的体验。
2. 不继续扩大 provider 类型，不做插件市场，不做桌面控制。
3. 真实 endpoint 的 API key 只能走环境变量或本地配置，不能写入 docs、日志或仓库。

### P3：真实语音输出

1. 接入真实 TTS provider。
2. 第一完整句可以先播，不等整段回复结束。
3. Stop 要同时停止 LLM stream、TTS queue、当前播放音频和嘴型。
4. 嘴型要来自实际播放或音频能量，不再依赖假音频。

### P4：语音输入

1. 麦克风选择。
2. VAD 触发录音边界。
3. ASR provider。
4. 语音输入必须复用同一条 runtime text path，不能绕开上下文、interrupt、错误恢复和 session 持久化。
